# 강남 테니스 봇 — "run failed" 메일 해결 설계

작성일: 2026-06-25

## 1. 문제 (증상)

- 봇이 5분마다 도는데, **가끔 GitHub에서 "run failed" 메일**이 온다.
- 실패 기록은 지금까지 **딱 4건**, 전부 6/24 13:35 ~ 6/25 02:05 사이, **모두 7분 30초 전후**로 종료됨.
- 정상 실행은 30~45초. 실패 실행만 7분+.

## 2. 원인 (근본 원인 — systematic-debugging으로 확정)

- 봇은 여러 예약 사이트(강남 24요청 + 송파/잠실 6요청 + 대치)에 **차례로 전화를 건다.**
- 각 요청은 `timeout 10~20초 × 재시도(최대 2번 더)` 구조 → 안 받는 곳 하나당 최대 1분씩 잡아먹음.
- 가끔 사이트가 느려지면 이 대기·재시도 시간이 **줄줄이 누적되어 7분을 초과**한다.
- 그러면 워크플로의 안전장치 `timeout-minutes: 7`이 봇을 **강제 종료** → GitHub이 "실패"로 기록 → **메일 발송.**
- 방아쇠: 6/24 커밋 `d0ff4f9 (재시도 2회 도입)` + 대치유수지 추가. 재시도가 각 요청 시간을 약 3배로 늘려, 코드 주석이 가정한 "최악 ~5.5분"을 실제로는 13분 이상으로 만들었다.
- **데이터/회사 장부와 무관. 강남 테니스 봇 자체 문제. 빈자리를 영구히 놓치는 것도 아님(다음 5분 회차가 회복).**

## 3. 해결 (두 갈래, 사용자 승인 완료)

### 3-1. 봇 빨리 끝내기 — "느린 사이트엔 빨리 포기하고 넘어가기"
| 항목 | 현재 | 변경 후 |
|---|---|---|
| 강남 기다림(timeout) | 10초 | **6초** |
| 송파·잠실 기다림 | 20초 | **8초** |
| 대치 기다림 | 20초 | **8초** |
| 재시도(안 받을 때 다시 걸기) | 2번 더 | **1번 더** |
| 재시도 간격(backoff_factor) | 1 | **0.5** |
| 봇 전체 최악 소요 | 약 13분 ❌ | 평소 30~45초 ✅ (극단 시 최대 ~7.5분 → 3-2 보험이 차단) |

→ 평소엔 7분보다 훨씬 빨리 끝난다. 단, 모든 사이트가 동시에 먹통인 극단에선 최대 약 7.5분
  (강남 24요청×6초×2시도 + esongpa 6요청×8초×2 + 대치·신청상태)으로 7분 가드를 살짝 넘길 수 있다.
  이 잔여 위험은 3-2의 보험(continue-on-error)이 받아 메일을 차단한다(두 장치가 함께 보장).
→ 밤 끊김 흡수용 재시도는 **1번은 유지**(완전히 없애지 않음). 재시도가 1회뿐이라 backoff(재시도 간 대기)는
  실제로는 적용되지 않는다(urllib3 특성) — 그만큼 더 빨리 끝나 안전 방향.

### 3-2. 보험 한 줄 — "7분 넘어도 실패로 안 침"
- `check.yml`, `daily-summary.yml`의 실행 step에 **`continue-on-error: true`** 추가.
- 만에 하나 다시 7분을 넘겨 강제 종료돼도, 워크플로를 "실패"로 기록하지 않음 → **메일 0.**
- 미래에 코트/사이트가 늘어도 메일이 안 오게 하는 보험.
- 봇 "완전 멈춤" 감시는 기존 healthchecks가 계속 담당.

## 4. 바뀌는 파일
- `src/http_session.py` — 재시도 파라미터(total/connect/read/backoff)
- `src/fetcher.py` / `src/esongpa.py` / `src/daechi.py` — 기다림(timeout) 값. 흩어진 숫자는 각 파일 상단 상수(`REQUEST_TIMEOUT`)로 모아 정리(DRY).
- `.github/workflows/check.yml` / `daily-summary.yml` — `continue-on-error: true` 한 줄
- `tests/test_http_session.py` — 재시도 횟수 검증을 1회 기준으로 갱신(TDD)

## 5. 안 하는 것 (YAGNI)
- 전역 워치독(signal.alarm), 사이트 병렬 조회 등 복잡한 구조 변경은 하지 않는다 — 위 두 갈래로 충분.

## 6. 검증
- `pytest`로 전체 테스트 통과 확인.
- 변경 요약 + 사용자 승인 후에만 main에 push(= 자동 실행 반영).
