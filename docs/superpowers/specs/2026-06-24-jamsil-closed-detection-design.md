# 잠실/송파 "접수시간 닫힘" 자동 감지 설계

> 날짜: 2026-06-24
> 상태: 설계 승인 완료 → 구현 진행

## 배경 (어제 디버그로 확정된 사실)

잠실 테니스장(esongpa)은 **하루 중 접수시간(낮, 대략 10~18시)에만 예약 접수**를 받는다.
그 외 시각(저녁·밤)에는 비어 있는 칸도 전부 `status_e`(접수불가시간)로 잠긴다.

2026-06-23 디버그 워크플로(`debug_jamsil.yml`) 실측:

| 조회 시각 | status_y(예약가능) | status_e(접수불가) | 비고 |
|---|---|---|---|
| 낮 12:59 (접수중) | 154 | 3 | 정상 — 빈칸 보임 |
| 저녁 21:18 (마감) | 0 | 151 | 전부 잠김 |
| 심야 00:49 (마감) | 0 | 146 | 전부 잠김 |

같은 `16:00~18:00` 코트가 낮엔 "예약가능", 저녁엔 "접수불가시간"으로 글자만 바뀐다.
즉 빈자리가 없어진 게 아니라 **접수창구가 닫힌 것**.

## 문제

현재 `src/esongpa.py`의 `_SLOT_RE`는 `status_y`(예약가능) 칸만 추출한다. 그래서:

1. **저녁 잠실 = 항상 0건** → 진짜 취소표(낮에만 뜸)와 무관하게 빈자리 없음으로 봄.
2. 낮에 잡았던 잠실 빈자리가 저녁에 0이 되면서 `has_changed`가 참 → **가짜 "현황 바뀜" 알림**.
3. 저녁의 0건이 `state.json`에 저장돼, 다음 낮에 빈자리가 다시 나타나면 또 변동 알림(출렁임).

## 해결: 데이터 기반 "닫힘" 감지 + 직전 유지

시각을 하드코딩하지 않고 **응답 데이터로 접수시간 여부를 판단**한다.
(접수시간 경계를 추측할 필요 없고, 시설이 시간을 바꿔도 자동 적응 — [[feedback_api_field_cross_check]] 교훈)

### 판단 규칙

한 시설(잠실/송파) 조회 결과에서 예약가능(`status_y`)·접수불가(`status_e`) 칸 수를 센다.

- **닫힘으로 판정**: 예약가능(`status_y`)이 0개이고 접수불가(`status_e`)가 **1개 이상**일 때. (예약가능이 하나라도 있으면 접수시간으로 보고 '열림')
  - 저녁: y=0, e=151 → 빈칸 100%가 접수불가 → 닫힘 ✓
  - 낮: y=154, e=3 → y>0이라 애초에 닫힘 아님 ✓
  - 진짜 다 참: y=0, e=0(빈칸 자체가 없음) → 닫힘 아님 → 빈자리 0 정상 처리 ✓

### 닫힘일 때 동작

그 시설은 **직전 상태(state.json)의 같은 시설 슬롯을 그대로 반환**한다(유지).
→ `has_changed`가 "변화 없음"으로 보고 저녁 내내 조용. 낮 접수시간이 되면 다시 정상 감시.

## 적용 범위

- **잠실 + 송파** 둘 다 동일 로직(같은 esongpa 시스템). 송파가 접수시간 제한이 없으면 이 장치는 발동하지 않음(무해·자동 보호).
- **강남(포이·세곡)** — REST API 방식이라 status 개념 없음. **영향 없음**.

## 바뀌는 파일

1. `src/esongpa.py`
   - 예약가능 슬롯 추출 외에 `status_e`/`status_y` 칸 수를 세는 로직 추가.
   - `fetch_esongpa_slots(settings, previous_slots=())` — 직전 기록을 인자로 받아, 닫힘 시설은 직전 슬롯으로 채움.
2. `src/main.py`
   - `run_vacancy_alert`/`run_summary`에서 직전 기록을 `fetch_esongpa_slots`에 넘김.
3. `tests/` — 위 3가지 상황(낮/저녁/다 참) + 직전 유지 동작 박제(어제 실측 숫자 사용).
4. **삭제**: `scripts/debug_jamsil.py`, `.github/workflows/debug_jamsil.yml` (매일 도는 검증용 cron 정지).
