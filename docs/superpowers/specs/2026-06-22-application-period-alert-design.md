# 신청기간 알림 기능 — 설계서

> 작성일: 2026-06-22
> 한 줄 요약: 포이·세곡 테니스장의 신청 접수가 열리는 순간(`준비중`→`접수중`)을 감지해
> 텔레그램으로 "신청 시작!" 알림을 보낸다. 기존 빈자리/취소표 알림에 더하는 기능.

> 기존 프로젝트(`gangnam-tennis-alert`)에 기능을 **추가**한다. 별도 시스템 아님.

---

## 1. 배경 — 왜 필요한가

조사 결과, 강남 테니스장은 **"심의승인" 방식**이다:
- 정해진 **신청기간(`receipt_period`)**에만 접수 → 심의·배정
- 시설마다 상태(`state_nm`)가 **`준비중`(접수 전) → `접수중`(접수 기간) → 마감** 으로 바뀜

기존 빈자리/취소표 알림(`state_cd 10/11` + `use_yn='N'`)은 **이미 `접수중`인 시설**(예: 세곡)에 작동한다.
하지만 **`준비중`인 시설**(예: 포이, 접수 6/24 시작)은 아직 예약이 안 열려 빈자리가 안 잡힌다.

→ **접수가 열리는 순간을 알려주면**(= "예약 일자 알림") 신청 시기를 놓치지 않는다.

---

## 2. 확정된 요구사항

| 항목 | 결정 |
|------|------|
| 대상 시설 | 포이 테니스장, 세곡 체육공원 (봉은은 범위 밖) |
| 알림 시점 | 상태가 **`준비중` → `접수중`**으로 바뀌는 순간 |
| 알림 도구 | 텔레그램 (기존과 동일) |
| 통합 | 기존 5분 주기 워크플로에 끼워넣음 (별도 시스템 X) |
| 기존 빈자리 알림 | **그대로 유지** (세곡 작동 중) |

---

## 3. 데이터 소스

**`rest/facilities/place_detail`** (GET, 파라미터 `company_code`·`part_code`·`place_code`)
응답 주요 필드:
- `state_nm` — 신청 상태: `준비중` / `접수중` / (마감 등)
- `receipt_period` — 접수 기간 (예: `2026-06-24~2026-06-29`)
- `period` — 이용 기간 (예: `2026-07-01~2026-07-31`)
- `confirm_type` — 예약 방식 (예: `심의승인`)

> 상태(`state_nm`)는 **시설 단위**라, 코트별로 같다. 그래서 시설마다 대표 코트 1개만 조회.

### 모니터링 대상 (실측 코드)
```
FACILITIES = [
  {name:"포이 테니스장",   comcd:"GNCC06", part:"04", place:"15"},
  {name:"세곡 체육공원",   comcd:"GNCC33", part:"04", place:"13"},
]
```

---

## 4. 구성 요소

1. **신청상태 조회기** — 각 시설의 `place_detail`에서 `{시설명, state_nm, receipt_period, period}`를 읽는다.
2. **전환 감지기** — 직전 상태와 비교해, `준비중` → `접수중`으로 바뀐 시설만 골라낸다.
3. **알림기(확장)** — 골라낸 시설을 "신청 시작!" 메시지로 만들어 텔레그램 발송.
4. **상태 저장(확장)** — 시설별 직전 `state_nm`을 파일에 보관(빈자리 기록과 별도 파일).

---

## 5. 동작 흐름 (5분마다, 기존 흐름에 추가)

1. **(기존)** 빈자리 조회 → 새 빈자리 → 알림
2. **(신규)** 시설 신청상태 조회 → `준비중`→`접수중` 전환 시설 → "신청 시작" 알림
3. 빈자리 기록 + 신청상태 기록을 각각 저장

> 첫 실행(직전 상태 없음)은 현재 상태만 저장(전환 알림 생략) — 알림 폭탄 방지.

---

## 6. 알림 메시지 형식

```
🎾 포이 테니스장 신청 시작!
📋 접수기간: 2026-06-24~2026-06-29
📅 이용: 2026-07-01~2026-07-31
👉 지금 신청: https://life.gangnam.go.kr/fmcs/1
```

---

## 7. 오류·예외 처리

| 상황 | 대응 |
|------|------|
| place_detail 조회 실패 | 그 시설만 건너뜀 (다른 시설·빈자리 알림은 계속) |
| 첫 실행(기록 없음) | 현재 상태만 저장, 전환 알림 생략 |
| 같은 전환 중복 알림 | 직전 상태와 비교하므로 한 번만 |

---

## 8. 범위 밖 (YAGNI)

- ❌ 봉은 테니스장 (코드 구조가 달라 별도 조사 필요 — 나중에)
- ❌ 접수 "마감" 알림 (시작만)
- ❌ 추첨/승인 결과 알림
- ❌ 신청기간 "하루 전" 예고 (사용자가 "시작 때 바로"만 원함)

## 9. 기존 코드와의 관계

- `src/fetcher.py` — 신청상태 조회 함수 추가 (`fetch_facility_status`)
- `src/config.py` — `FACILITIES` 목록 추가
- `src/state.py` — 시설 상태 저장/불러오기 추가
- `src/notifier.py` — 신청 시작 메시지 함수 추가
- `src/main.py` — 빈자리 알림 뒤에 신청상태 알림 추가
- `.github/workflows/check.yml` — 신청상태 기록 파일도 캐시에 포함
