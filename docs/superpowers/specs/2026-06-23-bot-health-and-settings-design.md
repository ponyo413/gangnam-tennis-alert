# 봇 고장 알림 + 설정표 직접 조정 — 설계서

> 작성일: 2026-06-23
> 한 줄 요약: ① 봇이 멈추거나 일부 실패하면 텔레그램으로 알림받기, ② 시설 켜고끄기·시간대를 코드 수정 없이 "설정표 파일"로 직접 조정하기.

---

## 배경 (왜 만드나)

현재 봇은 강남(포이·세곡)·송파·잠실 빈자리 알림(5분 주기) + 신청기간 알림 + 매일 아침 요약을 GitHub Actions(무료 클라우드)로 돌린다. 두 가지가 부족하다.

1. **봇이 조용히 멈춰도 사장님이 모른다.** 알림 0이 "자리가 없어서"인지 "봇이 죽어서"인지 구분 불가. 특히 GitHub의 무료 자동실행은 약 60일간 저장소 활동이 없으면 조용히 꺼지는 특성이 있어 "나도 모르게 멈춤"이 실제로 일어날 수 있다.
2. **시간대·시설을 바꾸려면 매번 코드를 고쳐 배포해야 한다.** 비개발자인 사장님이 직접 못 한다.

---

## 기능 ① 봇 고장 알림

### (A) 완전 멈춤 — 외부 감시 (dead man's switch)

죽은 봇은 스스로 "나 죽었어"를 못 보낸다. 그래서 **외부 무료 감시 서비스(healthchecks.io)** 가 봇의 생존 신호를 지켜본다.

- 봇 워크플로(`check.yml`) 끝에 **"살아있어" 신호(고유 URL로 GET 요청) 한 줄**을 추가한다.
- healthchecks.io 설정: 기대 주기 5분 + 유예시간. **약 10분 넘게 신호가 끊기면 알림 발동.**
- 알림 경로: healthchecks.io → **사장님 텔레그램**(지금 쓰는 봇/채팅 그대로 재사용). 무료.
  - 구현은 healthchecks.io의 webhook에 텔레그램 `sendMessage` 호출을 넣는 방식(봇 토큰 재사용).
- 효과: **GitHub Actions가 통째로 멈춰도**(자동 비활성화·장애 등) 외부에서 감지해 알린다.
- 사용자 1회 작업: healthchecks.io 가입 + 텔레그램 연동(작업 시 화면 보며 안내).

### (B) 일부 실패 — 하루 한 번

봇은 살아있는데 한 시설만 막히는 경우(예: 잠실 로그인 실패)는 봇이 직접 알릴 수 있다. 단 5분마다 반복하면 시끄러우니 **하루 한 번**으로 묶는다.

- 조회 중 시설별 실패를 **기록(누적 카운트, state 파일)**.
- **매일 아침 8시 요약** 발송 시 직전 24시간 실패를 `⚠️ 어제 잠실 조회 3번 실패` 형태로 **요약에 한 줄** 덧붙인다.
- 발송 후 카운트 리셋.

## 기능 ② 시간대·시설 직접 조정

### 설정표 파일 (`settings.yaml`)

사람이 읽기 쉬운 형식. 사장님이 GitHub 웹 화면에서 편집 → **다음 점검(최대 5분)부터 반영.**

```yaml
# ───────────────────────────────────────
#  🎾 테니스 알림 설정표
#  이 표를 고치면 다음 점검(최대 5분)부터 반영됩니다.
#  · 받기: true(알림 받기) / false(끄기)
#  · 시간: 시작 시각 숫자만 — 20 = 저녁 8시
# ───────────────────────────────────────

강남:                      # 포이·세곡
  받기: true
  매일: [19, 20, 21]       # 저녁 7·8·9시

송파:
  받기: true
  토: [8, 10]              # 토요일 오전 8·10시

잠실:
  받기: true               # ← 잠깐 끄려면 false 로만 바꾸면 끝
  토: [18, 20]
  일: [18, 20]
  월: [20]
  화: [20]
  수: [20]
```

- **시설 켜고끄기:** `받기: true ↔ false`
- **시간대:** 요일별 시작시각 목록(`매일`은 모든 요일 공통)
- **사용 예:** 잠실 목요일 추가 → `목: [20]` 한 줄 추가 / 강남 시간 변경 → 숫자 수정

### 읽기 · 검증 · 폴백 (안전장치)

- 봇이 **매 실행 시 `settings.yaml`을 먼저 읽어** 그대로 적용.
- **형식 검증:** 시설명·받기·시간 형식 확인.
- **형식 오류 시:** 텔레그램 `⚠️ 설정표 형식 오류` 알림 + **직전 정상 설정으로 계속 작동**(봇 안 멈춤).
- 정상 읽기 성공 시 그 설정을 "마지막 정상 설정"으로 백업.
- 설정표가 아예 없으면 코드에 내장된 **기본값**으로 작동.

### 적용 범위

- **시간대 필터**(`filters`): 설정표의 시설별 요일/시간 기준으로 판별.
- **시설 on/off:** `받기: false`면 그 시설 조회 자체를 건너뜀.

## 바뀌는 파일

| 파일 | 변경 |
|---|---|
| `settings.yaml` (신규) | 사장님이 고치는 설정표 |
| `src/settings_loader.py` (신규) | 설정표 읽기 + 형식 검증 + 폴백 + 기본값 |
| `src/filters.py` | 시간대 판별을 설정표 기준으로 |
| `src/main.py` | 설정 읽어 적용, 시설 on/off, 실패 기록, 요약에 실패 보고 |
| `src/notifier.py` | 요약 메시지에 "어제 실패" 줄 추가 |
| `src/state.py` | 실패 카운트 저장/불러오기 |
| `.github/workflows/check.yml` | 끝에 "살아있어" 신호(healthchecks 핑) 한 줄 |
| `requirements.txt` | YAML 읽기 라이브러리(PyYAML) 추가 |
| `tests/` | 설정 읽기·검증·폴백·시간대 적용·실패 보고 테스트 |

> **봇 본체(빈자리 읽기·비교·발송·신청기간)는 그대로.** "설정을 어디서 읽나"와 "고장 알림"만 더한다.

## 안전 / 되돌리기 계획

- **데이터 안전:** 개인용 별도 repo, DB 없음. 읽기 전용 조회 + 텔레그램 발송뿐.
- **봇 보호:** 설정표 실수는 폴백으로 막고, 봇 본체 로직은 손대지 않음(회귀 테스트로 보호).
- **되돌리기:** 모든 변경은 git 커밋 단위 → 문제 시 `git revert`. 설정표를 지우면 코드 내장 기본값으로 작동.

---

## 범위에서 제외 (YAGNI)

- **텔레그램 양방향 명령** — 설정표로 충분(상시 서버 없는 구조라 과한 개발).
- **새 시설/종목 추가** — 파싱이 필요해 별도 작업(설정표로 on/off만, 추가는 코드).
- **잠실 접수시간 이슈**(저녁 예약불가) — 별개로 진행(오늘 저녁 확인 후).

---

## 구현 순서 (개략 — 상세는 공정표)

1. 설정표 읽기·검증·폴백·기본값 모듈 + 테스트
2. `filters`·`main`이 설정표를 쓰게 전환 + 회귀 테스트(기존 동작 보존)
3. `settings.yaml` 실물 작성(현재 시간대와 동일하게 시작)
4. 일부 실패 기록 + 아침 요약 보고 + 테스트
5. `check.yml`에 healthchecks "살아있어" 신호 추가
6. healthchecks.io 가입·텔레그램 연동 안내(사용자 1회)
7. 전체 검증 → 사용자 승인 → 1층(main) 배포
